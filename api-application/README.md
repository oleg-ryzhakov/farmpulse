# API приложения (мобильный клиент)

HTTP-эндпоинты для **владельца** и приложения. Отдельно от **приёма данных от ферм** (`../api/worker/`).

Контракт и потоки: [`../docs/MOBILE-CLIENT-SERVER.md`](../docs/MOBILE-CLIENT-SERVER.md).

---

## Где сейчас хранятся данные о фермах и статусе

| Что | Где |
|-----|-----|
| Файл | `api/v2/farms/config.json` (рядом с `config.json.example`) |
| Формат | Один JSON: корень `farms`, внутри ключи — строковые id ферм |

Поля по ферме (фактически используемые для «текущего состояния»):

| Поле | Смысл |
|------|--------|
| `status` | Строка, в worker при контакте выставляется в **`online`**. |
| `last_seen_at` | UTC `Y-m-d H:i:s`, обновляется при **`hello`**, **`stats`**, **`message`** в `api/worker/api.php`. |
| `gpu_temps` | Массив температур (обновляется в основном из **`stats`**). |
| `gpu_count` | Число GPU (из stats/hello). |
| `name`, `password`, `token`, `commands`, … | Конфигурация и очередь команд — не «метрики в реальном времени». |

Кто **пишет** статус и last_seen: **`api/worker/api.php`** при успешных запросах от рига (не веб-панель).

Кто **читает** для списка: **`api/v2/farms/farms.php`** (`GET` — отдаёт `farms[]` с `status`, `last_seen_at`, `gpu_temps`, …).

**Важно:** перевод в **`offline` при отсутствии heartbeat** в текущем PHP **не делается** — в коде `offline` задаётся при **создании** фермы (`farms.php` `POST`) и дальше worker только поднимает `online`. Актуальность «жива ли ферма» по смыслу нужно проверять по **`last_seen_at`** и порогу времени (отдельный cron/скрипт — позже, или в `api-application`).
