# План архитектуры: единый сервис управления фермами

> Рабочая копия. Родительский проект-заготовка: каталог `../` (hive-management).

## Цель

Один VPS с центральным сервисом: статистика, алерты, опционально управление питанием и мобильное приложение — без смены «родного» URL HiveOS API в настройках рига (через промежуточный скрипт/агент).

---

## Слои (сверху вниз)

### 1. Риги HiveOS (без смены стандартного API URL)

- В HiveOS остаётся **официальный** endpoint Hive (как сейчас у вас в образе).
- Рядом работает **ваш лёгкий клиент** (cron/systemd на риге или отдельный контейнер): он **читает** локальный статус / дублирует heartbeat и **отправляет** данные на ваш VPS (`POST` с `rig_id`, токеном, метриками).
- *Альтернатива (сложнее):* reverse-proxy только если технически допустимо; вы явно предпочитаете **не трогать URL в OS** — значит приоритет у **клиента на риге** или у **сбора с вашего текущего worker**, если риг уже смотрит на ваш сервер.

### 2. VPS — единый backend

| Компонент | Назначение |
|-----------|------------|
| **API** | Приём данных от ригов/прокси, отдача в мобильное приложение; учёт ферм, аккаунтов Hive, привязок «ферма ↔ розетка». |
| **Хранилище** | Состояние ферм, last_seen, история метрик, настройки алертов, привязки Sonoff/Ewelink. |
| **Мониторинг offline** | Правило: нет данных N минут → событие «offline». |
| **Telegram** | Бот: уведомления о offline/online, по желанию подтверждение действий. |
| **Интеграция Wi‑Fi реле** | Sonoff / eWeLink (или другой облачный API): привязка `farm_id` → `device_id`; команды **вкл/выкл** по API; сценарий «offline долго → уведомление → ручное или авто power cycle». |
| **Веб-панель** (опционально) | То, что у вас уже есть на PHP, можно постепенно перенести или оставить как тонкий UI к тому же API. |

### 3. Промежуточный «свой API скрипт» (смысл)

- Не подменять URL в HiveOS, а **синхронизировать**: скрипт на вашей стороне (на VPS или на риге) **пересылает/агрегирует** состояние в единый сервис.
- Если риг **уже** шлёт на ваш `worker` — этот поток остаётся; новый сервис **подписывается** на те же события или забирает из БД.

### 4. Android-приложение

- Один экран списка ферм (все аккаунты Hive **логически** сведены у вас в сервисе).
- Детали: температура, last seen, кнопки «перезагрузка через Hive», «питание Sonoff» (если разрешено политикой безопасности).
- Авторизация: токен к вашему API (не пароли Hive в приложении — только к вашему бэкенду).

---

## Потоки данных (кратко)

```
Риг → (клиент/cron ИЛИ ваш текущий worker) → VPS API → БД
                                              ↓
                                        Telegram / eWeLink / Android
```

---

## Безопасность (минимум)

- Токены на ферму, HTTPS на VPS.
- Команды на розетку — только после явного действия или строгих правил (авто power cycle — осторожно, риск циклов).

---

## Этапы внедрения (логичный порядок)

1. VPS + API + БД + приём данных от существующего worker/клиента.  
2. Единая модель «ферма» + offline + Telegram.  
3. Привязка Sonoff/eWeLink и кнопки питания.  
4. Android к тому же API.  

---

## Связь с заготовкой в родительском репозитории

- В этом репозитории: **`api/worker/api.php`**, **`api/v2/farms/config.json`** — приём от рига; **`web/`** — веб-морда отдельно от API.
- В родителе `../` — старая смешанная вёрстка в одном `public/`; здесь разделено на **`web/`** + **`api/`**.
