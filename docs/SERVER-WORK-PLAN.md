# План работ: новая серверная часть на чистой Ubuntu (заготовка — родительский проект)

> Рабочая копия. Исходные файлы заготовки: каталог `../` (hive-management).

> **Деплой с этого репозитория:** каталог **`web/`** — веб-морда; каталог **`api/`** — только API (worker, farms, market). Nginx: `root` → `web`, `location /api/` → `alias` на `api`. Перед первым запуском скопировать `api/v2/farms/config.json.example` → `config.json`.

Цель: на VPS с **чистой Ubuntu** развернуть **веб-интерфейс** (можно копировать текущий) и **API для ригов** — приём `hello` / `stats` / `message`, отдача команд из очереди. Ниже — порядок работ и соответствие файлам в родительском репозитории.

---

## Что взять из текущей заготовки

Пути ниже относительно **`../`** (корень hive-management).

| Роль | Файлы/папки (источник) |
|------|-------------------------|
| **Worker API (риги)** | `../api/worker/api.php` — логика `hello`, `stats`, `message`, очередь `commands`, пароль из `config.json` |
| **Конфиг ферм** | `../api/v2/farms/config.json` — структура `farms`, пароли, `gpu_temps`, `commands` |
| **Управление с UI** | `../api/v2/farms/farms.php`, `workers.php`, `command.php`, `flight.php` |
| **Диспетчер API** | `../api/index.php`, `../api/.htaccess` — под Apache; на nginx правила переписываются |
| **Фронт** | `../index.php`, `../farm.php` — Bootstrap, опрос `api/v2/farms/*.php` |
| **Опционально позже** | `../api/v2/market/*` — маркет Hive; на первом этапе можно **не копировать**, если вебу хватит ферм без вкладки Flightsheets из маркета |

## Этап 0. Базовая система (Ubuntu)

1. Обновление пакетов, часовой пояс, при необходимости локаль `UTF-8`.
2. Пользователь для деплоя (не root), SSH-ключи, отключить пароль root по SSH при желании.
3. Файрвол: `22` (SSH), `80`/`443` (HTTP/HTTPS).

---

## Этап 1. Веб-стек под PHP (как у заготовки)

Текущий фронт и API — **PHP**, без фреймворка. На чистой Ubuntu разумный минимум:

1. **nginx** — раздача статики и `try_files` → `index.php`.
2. **PHP-FPM** (версия 8.x) с расширениями: **json**, **curl**, **mbstring** (для `pools.php` и строк).
3. Каталог сайта, например `/var/www/hive-management`, права на запись только туда, где нужны логи/кэш (см. этап 4).

*Альтернатива:* Apache + mod_php — ближе к OSPanel, но на VPS чаще nginx + php-fpm.

---

## Этап 2. Развёртывание кода с заготовки

1. Скопировать на сервер **минимальный набор** из `../`:
   - корень: `index.php`, `farm.php`
   - `api/worker/api.php`
   - `api/index.php`, `api/.htaccess` (для nginx — см. этап 3)
   - `api/v2/farms/` — `farms.php`, `workers.php`, `command.php`, `flight.php`, пустой/шаблонный `config.json`
   - при необходимости UI с маркетом: `api/v2/market/*` + кэш-файлы или права на запись.
2. **Не копировать** на прод без необходимости: `api/v2/requests.log`, старые `*.cache.json` с секретами, бэкапы расширений Chrome.
3. Создать **свежий** `config.json` с тестовой фермой или импортировать структуру вручную.

---

## Этап 3. Маршрутизация (критично для ригов)

Риги обращаются к worker примерно так:  
`/api/worker/api.php?id_rig=…&method=hello|stats|message`

Нужно:

1. Location в nginx для `/api/worker/api.php` → PHP-FPM (или единый `location ~ \.php$`).
2. Аналог правил из `api/.htaccess`: запросы без реального файла — на `api/index.php`, **если** вы используете диспетчер для части путей.
3. Проверить с VPS: `curl` к `hello` с правильным `id_rig` и паролем из тела (как в текущем `api.php`).

---

## Этап 4. Данные и права

1. `config.json` — владелец `www-data` (или пользователь php-fpm), права **640** (чтобы не светить пароли миру).
2. Если оставляете логи как в заготовке — каталог для `requests.log` с записью только для веб-пользователя.
3. Кэши маркета (`pools.cache.json` и т.д.) — отдельная подпапка с записью, если маркет включён.

---

## Этап 5. HTTPS (рекомендуется до подключения реальных ригов)

1. **Certbot** (Let's Encrypt) для вашего домена.
2. Редирект HTTP → HTTPS.
3. В конфиге рига / в `hello` убедиться, что `HIVE_HOST_URL` или аналог указывает на **https://**.

---

## Этап 6. Проверка сценариев (приём / команды)

По логике `api/worker/api.php`:

| Действие | Проверка |
|----------|----------|
| Регистрация фермы в UI | POST в `farms.php` → запись в `config.json` |
| `hello` | 200, пароль совпадает, в ответе есть `config` |
| `stats` | Обновляются `last_seen_at`, `gpu_temps` |
| Очередь команды | Из UI `command.php` → `reboot` → в следующем `stats`/`message` риг получает команду |
| Веб | `index.php` показывает online/offline по `last_seen` (порог ~120 с; UTC; интервал sidecar обычно 30 с) |

---

## Этап 7. Фронт: что подправить после копирования

1. Базовые URL: если сайт не в корне домена, вынести префикс в константу или `<base href="...">`.
2. Убрать или заменить обращения к несуществующим элементам (в текущем `index.php` встречается ссылка на `fsCoinOverride` — при отсутствии поля вкладка Flightsheets может требовать мелкой правки).
3. Вкладка **Flightsheets** зависит от `api/v2/market/*` — на первом этапе можно скрыть вкладку или оставить только карточку фермы в `farm.php`.

---

## Этап 8. После стабилизации (не блокирует запуск)

1. Вынести `config.json` в **SQLite/PostgreSQL** — одна БД для ферм, истории метрик, пользователей веб-панели.
2. Отдельный процесс для **offline + Telegram** (cron или worker), как в `ARCHITECTURE-PLAN.md`.
3. JWT/сессии для веб-API вместо открытых эндпоинтов `farms.php` (сейчас без авторизации пользователя панели).

- [ ] nginx + php-fpm, сайт открывается по HTTPS  
- [ ] Скопированы worker + farms API + `index.php` / `farm.php`  
- [ ] `config.json` создан, права безопасны  
- [ ] `curl` имитирует `hello` и `stats` с тестовой фермой

## Зависимость от корня веб-приложения

Пути в JS вида `api/v2/farms/farms.php` предполагают, что **корень веб-сервера** — тот же каталог, где лежат `index.php` и папка `api/`. При другом layout нужен общий префикс в JS.
