# Мобильный клиент ↔ сервер (контур владельца)

Документ задаёт **механизм работы приложения владельца ферм**. Он **не** является продолжением и **не** дублирует контракт **приёма данных от ферм** (`api/worker/`, Hive-совместимый поток).

---

## 1. Два контура API

| Контур | Клиент | Назначение |
|--------|--------|------------|
| **Ingest (фермы)** | Риг / агент, `worker`-совместимые запросы | Периодическая отправка телеметрии, получение конфигов, команды для майнера. Аутентификация: токен/пароль **фермы** в конфиге сервиса. |
| **Owner (владелец)** | Android-приложение, веб-панель (опционально) | Просмотр агрегированного состояния, алерты, управление в рамках политики. Аутентификация: **учётная запись владельца** (или выданный сервером токен с привязкой к списку `farm_id`). |

Приложение **никогда** не выступает как «ещё одна ферма» и **не** использует те же эндпоинты и семантику, что поток от ригов к `worker`.

---

## 2. Источник истины и задержки

- Фермы отдают данные на сервис с интервалом **10–30 с** (фактический период задаётся на стороне рига/агента).
- Сервер хранит **последнее известное состояние** и вычисляет **производные** величины: `last_seen`, `online`/`offline` по правилу таймаута (нет данных дольше `T` после ожидаемого периода).
- **Точность по времени события на ферме** ограничена периодом опроса и правилом offline на сервере.
- **«Реальное время» в UI** означает: после обновления состояния на сервере клиент получает его **без дополнительной пользовательской задержки** (см. каналы ниже), а не «быстрее, чем ферма шлёт данные».

---

## 3. Каналы: приложение ↔ сервер

| Канал | Роль |
|-------|------|
| **HTTPS + JSON (REST)** | Логин владельца, список ферм, карточка фермы, регистрация устройства для push, настройки уведомлений. |
| **Доставка событий в фоне** | **FCM**: сервер инициирует push при смене критичного состояния (например переход в `offline`), а не при каждом heartbeat фермы. |
| **Живое обновление при открытом приложении** | **WebSocket** или **SSE** с сервера: события «снимок/патч по фермам изменился» **после** записи в БД; либо короткий polling с `ETag`/`If-Modified-Since` на REST — проще, хуже по задержке UI. |

Клиент **не** опрашивает фермы напрямую. Все данные — только из **Owner API** и push от **вашего бэкенда**.

---

## 4. Серверная логика (обязательная для push)

1. Ingest обновляет строки состояния ферм / last_seen.
2. Фоновая задача или триггер сравнивает предыдущий и новый агрегированный статус (например `online` → `offline`).
3. При срабатывании правила создаётся **событие** (лог/очередь) и вызывается **отправка FCM** на токены устройств данного владельца (фильтр по подписке/типу события).

Детекция offline выполняется **на сервере**; приложение лишь отображает результат и реагирует на push.

---

## 5. Минимальный контракт Owner API (целевой)

Префикс и имена — пример; важно разделение **сессии владельца** и **токена фермы** в Ingest.

| Метод | Назначение |
|-------|------------|
| `POST /owner/auth/login` | Логин/пароль или OAuth-провайдер → `access_token` + метаданные пользователя. |
| `GET /owner/farms` | Список ферм владельца: `id`, имя, вычисленный статус, `last_seen`, агрегаты (температуры, хешрейт — по наличию в модели). |
| `GET /owner/farms/{id}` | Детали одной фермы. |
| `POST /owner/devices` | Регистрация **FCM token** + платформа + опционально список подписок на типы событий. |
| `DELETE /owner/devices/{id}` | Отзыв токена при выходе или смене устройства. |

Опционально: `GET /owner/stream` (WebSocket upgrade) или отдельный URL для SSE — события только после коммита состояния на сервере.

Заголовок: `Authorization: Bearer <access_token>` для всех запросов кроме login.

---

## 6. Что клиент не делает

- Не вызывает `api/worker/` и пути, предназначенные для ригов.
- Не считает оффлайн только по локальному таймеру без сервера (максимум — оптимистичный UI до ответа API).
- Не хранит пароли Hive; допустимо только хранение сессии/токена к **Owner API** в защищённом хранилище ОС.

---

## 7. Связь с текущим кодом в репозитории

- **Ingest**: `api/worker/api.php`, обновление `config.json` / состояния ферм по мере реализации.
- **Прототип списка ферм без разделения владельца**: `api/v2/farms/farms.php`, `api/v2/auth/login.php` (сейчас — уровень **фермы**, не аккаунта владельца). Целевой **Owner API** вводится как отдельный namespace и модель пользователя либо эволюционирует из веб-панели по единому контракту.
